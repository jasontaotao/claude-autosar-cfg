01

入门Getting Started

1.1 产品定位

claude-AutosarCfg 是一款独立的桌面 GUI， 面向 AUTOSAR Classic Platform 的 BSW（Basic Software）配置工程师。 它在 ARXML 解析器与 ECU 抽取（ECU Extract）生成器之间， 提供了图形化、实时校验、可视化引用追踪的中间编辑层。

1.2 核心能力一览

  • ARXML IO · v0.2.0 原生 Electron 对话框打开 / 保存 .arxml；支持 r4.x ECUC 子集，15+ fixture 文件 round-trip。
  • 树形导航 + 7 种参数编辑器 · v0.3.0 左：递归 ARIA 树（package → module → container → param）。右：类型感知编辑器（string/integer/float/boolean/enum/reference/multiline）。
  • 实时校验（7 种 kind）· v0.4 – v0.8 每次编辑同步触发校验：rangeenumreferencerequiredschemamultiplicitycross-ref。
  • BSWMD 模块管理 · v1.1.0 从 BSWMD schema 模板一键生成 ECUC 容器实例（Skeleton + 默认值）。支持 chip 多选 + dirty-guard。
  • 多文件 ECUC 合并导入 · v1.2.0 EB tresos 风格的 Resolve Conflicts wizard：按 module 维度聚合多源，撞名 diff 三栏 + 原子提交 + 单步撤销。

1.3 系统要求

# 2. 一次性构建 main / preload 进程

# 3. 启动开发模式

#    - main / preload 修改后需重启 pnpm dev

2.4 状态指示符规范

应用使用统一的状态色，跨所有 panel 一致：

  • Saved / Valid 文件已落盘；校验全部通过；schema 解析成功。
  • Unsaved / Warning 有未保存编辑；schema 警告；BSWMD 缺失引用。
  • Error / Blocked 校验失败；引用解析失败；save 写入失败。
  • Loading / Pending IPC 在飞；commit 在执行；debounced 校验排队中。
03

项目管理Project Management

一个 claude-AutosarCfg Project 是同一 BSW 工程在磁盘上的逻辑容器： manifest.json 记录所有 value-side ARXML、schema-side BSWMD 引用与项目元数据。项目根目录下不会写额外二进制， 所有编辑最终落回原 ARXML 文件。

3.1 manifest.json 内存布局

  "schemaVersion": 2,
  "name": "S32K148_EAS_EB_3399A",
  "createdAt": "2026-06-13T08:14:22Z",
  "valueArxmlPaths": [
    "Com_Com_EcucValues.arxml",
    "Det_Det_EcucValues.arxml",
    "EcuC_EcuC_EcucValues.arxml",
    "PduR_PduR_EcucValues.arxml",
    "WdgIf_WdgIf_EcucValues.arxml"
  ],
  "bswmdPaths": [
    "bswmd/Com_Bswmd.arxml",
    "bswmd/Det_Bswmd.arxml",
    "bswmd/EcuC_Bswmd.arxml",
    "bswmd/PduR_Bswmd.arxml",
    "bswmd/WdgIf_Bswmd.arxml"
  ]
}
i
路径以 manifest 相对路径 持久化（v1.1.1+）。v1.1.0 时代的绝对路径会在 loadManifest() 时自动 migrateManifestPaths()，对用户透明。

3.2 新建项目

菜单 Project → New Project… 或顶部 AppHeader 的 Project 下拉触发 NewProjectDialog，它是一个统一入口，整合了模板选择、目录挑选与 BSWMD chip 多选（v1.1.0 引入）。

3.3 新建流程

  1. 输入项目名 S32K148_EAS_EB_3399A，遵循 ECU 工程命名惯例（kebab-case 或 PascalCase，禁止路径分隔符）。
  2. 选择目标目录 点击 Browse 调起原生文件夹选择器，确认后回填绝对路径。
  3. 选择模板 Empty（完全空白）/Classic（带 Classic 5 模块脚手架）/Clone（从既有项目复制 ARXML + BSWMD）。
  4. 勾选 BSWMD 模板为 Classic 时，BSWMD chips 会预填 5 个标准 BSW 模块；可继续添加自定义 BSWMD。
  5. 点击 Create 若有未保存编辑，会先弹 ConfirmDialog（3 选项：Save / Discard / Cancel）。确认后 IPC: project:new 落盘 manifest + 模板文件。
  6. 切换视图 项目创建后自动 openProject()，新文件立即出现在 LeftPanel 的 Files 标签。

3.4 打开项目

两种方式：

  • 菜单入口：Project → Open Project… 选 manifest.json。
  • 直接双击：在文件管理器双击 manifest.json， 系统需先建立文件关联（NSIS 安装包已自动注册 .autosarcfg 关联）。
  • Recent 列表：Project → Recent 展开最近 8 个项目。
!
打开时若当前有 dirty 编辑，会先弹 ConfirmDialog 询问处置。 注意：关闭项目 不会自动保存，请先 Save All。
04

ARXML 文件操作ARXML File Operations

一切配置都源于 ARXML（AUTOSAR XML）。本章说明 claude-AutosarCfg 如何打开、编辑、保存这类文件， 以及背后的 IPC 通道、错误处理与版本兼容策略。

4.1 解析能力

解析器（core/arxml/parser.ts）支持 AUTOSAR r4.x 的 ECUC 子集：

元素 覆盖 说明
AR-PACKAGE 递归 支持任意嵌套深度，递归下降为树节点。
ELEMENT (module / container) ✓ 通过 SHORT-NAME + DEFINITION-REF 标识。
PARAMETER-VALUES ✓ 包装数值参数；按 DEFINITION-REF DEST 决定类型。
REFERENCE-VALUES ✓ (v0.8+) 标准 <REFERENCE-VALUES> 包装 + EcuC vendor dialect。
VALUE-REF DEST ✓ 跨模块引用；路径索引全局可见。
CHOICES / LITERALS ✓ enum 字面量集合。
MULTIPLICITY ✓ (v0.6+) 容器子实例上下界。
SW-BASE-TYPE 等 C-impl 字段 ⏸ Pass-through 解析保留原样，UI 不暴露。

4.2 打开 ARXML

  1. 触发入口 File → Open ARXML… 或 Ctrl+O 唤起原生文件对话框。
  2. 选择文件 一次可多选（按住 Shift / Ctrl）；多文件会进入 Combined Tree 模式。
  3. 解析与 schema 匹配 arxml:parse IPC 启动 fast-xml-parser；schema 侧通过 buildPathIndex + extractReferences 建立全局索引。
  4. 首次校验 同步运行 validateProject，耗时通常 < 50ms（5 fixture 9.2 MB）。
  5. 渲染 树与编辑器挂载；校验结果分组显示。

4.3 保存 ARXML

保存路径（arxml:save）由 saveArxmlHandler 处理。v1.1.2 起错误按 NodeJS errno code 区分 6 种 typed kind：

errno Typed kind 触发场景 用户提示
EACCES read-only 文件被设为只读 / 持有者无写权限 "文件被锁定，请检查权限"
EPERM forbidden Windows ACL / SELinux 拒绝 "权限被拒绝，请联系管理员"
ENOSPC no-space 磁盘满 "磁盘空间不足，请清理"
EDQUOT quota 用户配额超限 "磁盘配额超限"
ENOENT missing 父目录不存在 / 已被删 "目录不存在"
ENOTDIR not-dir 路径中段不是目录 "路径无效"
EIO / 其它 write-failed 通用 IO 错误 "写入失败：<原因>"

4.4 多文件 Save All

当 multi-ECUC dirty session 激活时（v1.1.1+），AppHeader 的 Save 按钮旁出现 Save All。 每个文件独立 partial-failure UI：

参数编辑Parameter Editor

右侧 ParamEditor 根据当前选中节点的 DEFINITION-REF 决定渲染哪一类输入控件。 7 种类型与 BSWMD schema 一一对应，每种都有独立的 <ModeName>Editor.tsx：

6.1 7 种参数类型

类型 DEFINITION-REF DEST 控件 校验
string ECUC-STRING-PARAM-DEF 单行 input maxLength (range kind)
integer ECUC-INTEGER-PARAM-DEF numeric input（step=1） min/max (range kind)
float ECUC-FLOAT-PARAM-DEF numeric input（step=any） min/max (range kind)
boolean ECUC-BOOLEAN-PARAM-DEF checkbox / toggle —
enum ECUC-ENUMERATION-PARAM-DEF 原生 <select> enum value (enum kind)
reference ECUC-REFERENCE-VALUE / ECUC-FOREIGN-REFERENCE-DEF text + DEST badge + picker path resolves? (reference / cross-ref)
multiline ECUC-MULTILINE-STRING-PARAM-DEF textarea maxLength (range kind)

6.3 编辑流程

  1. 选中树节点 点击 ECUC TREE 中任一节点，ParamEditor 加载该节点的参数定义。
  2. 字段渲染 ParamEditor 按 DEFINITION-REF DEST 决定控件类型，调用对应 <Mode>Editor。
  3. 用户输入 所有变更经 useArxmlStore.updateParam(path, key, value) 进入 store。
  4. 同步触发校验 store action 同步调用 validate(documents)，错误的参数行内显示红色边框。
  5. 300ms debounced revalidate useDebouncedValidation(300) 兜底，覆盖未来可能的 async mutation 路径。
  6. Dirty 标记 store 检测 mutation，自动设 isDirty = true，AppHeader 的 Save 按钮变橙。

6.4 Reference 编辑器深度

引用是最复杂的类型。ReferenceEditor 支持：

  • 实时校验路径存在性：每输入一个字符，跨全部 document 查 buildPathIndex，匹配时高亮 ✓ resolves，未匹配时标红 ✕ broken。
  • DEST 约束：DEST badge 显示 ECUC-REFERENCE-VALUE 等目标定义，类型不匹配时提示。
  • Picker 唤起：右侧 📁 按钮唤起 ModuleFromBswmdPicker，按 BSWMD schema 过滤候选。
  • 占位跳过：空路径或 / 视为未设置（isUnsetPlaceholder），不进入 cross-ref 校验。
✓
v0.8 重大改进：parser 同时支持标准 <REFERENCE-VALUES> 包装 （Com / PduR / WdgIf）与 EcuC vendor dialect（<REFERENCE-VALUE> 嵌套在 <PARAMETER-VALUES> 下 + DEST="ECUC-FOREIGN-REFERENCE-DEF"）。 序列化时统一输出标准格式，跨方言 round-trip 字段级一致。
07

实时校验Live Validation

校验是 claude-AutosarCfg 的灵魂功能。每次按键都同步触发 validateProject(documents)，把违规即时反馈到 UI。 7 种 ValidationErrorKind 覆盖 BSW 工程 95% 的常见问题。

7.1 7 种 Validation Kind 色卡

Kind 颜色 触发条件 示例
range 🔴 红 数值 / 字符串越界 ComIPduLength=70000 超 max=65535
enum 🟠 桃 枚举值不在 CHOICES ComTxMode=DIRECTD（拼错）
reference 🟡 黄 本文件内引用路径不解析 VALUE-REF 指向不存在的 path
required 🟢 绿 必填参数缺失 ComIPdu.ComIPduHandleId 未填
schema 🟣 薰衣草 BSWMD 未提供该定义 参数不在 ECUC schema 中
multiplicity 🔵 蓝 container 直接子实例数违反 [lower, upper] 某 ComIPduGroup 需要 ≥1，0 个
cross-ref 🟦 青 跨文档引用目标不在 path index 引用了未加载模块的 path

跨容器引用Cross-Container References

AUTOSAR 工程的核心特征是跨容器引用：一个 ComIPdu 引用 ComSignal， PduR 的路由表引用 ComIPdu 的 handle，Det 的错误源引用任意模块的 API。 claude-AutosarCfg 用全局 path index 统一管理这些引用，校验时跨文档比对。

8.1 引用模型

// core/arxml/types.ts
interface ParamValue {
  type: 'reference';
  value: string;       // path: "/Com/ComConfig/ComSignal/..."
  dest?: string;       // ECUC-REFERENCE-VALUE / ECUC-FOREIGN-REFERENCE-DEF
}

8.2 全局 path index

每次 ARXML 加载后，buildPathIndex(documents) 遍历所有元素， 以完整路径为 key 索引到节点引用。索引在内存中是 Map<string, ElementRef>： 

// 索引条目示例（v0.8.0 跑 S32K148 fixture 后）
{
  "/Com/ComConfig/ComIPdu/ComIPdu_001": { docId, kind: 'container', shortName: 'ComIPdu_001' },
  "/Com/ComConfig/ComSignal/ComSignal_EngineSpeed": { ... },
  "/PduR/PduRRoutingTable/RoutingTable_0/...": { ... }
  // ... 共 14,200+ 条目
}

8.3 跨引用解析流程

  1. 扫描所有 VALUE-REF extractReferences(documents) 收集每个 reference 类型参数的 { value, dest }，返回 readonly ReferenceSite[]。
  2. 逐条 path index 查找 checkCrossRefs(sites, index) 对每条 site 调 index.get(path)。
  3. emit ValidationError 缺失时 emit { kind: 'cross-ref', path: sourcePath, message: 'unresolved' }。
  4. 聚合入 project 结果 validateProject(docs) 把 cross-ref 与其它 6 种合并后写入 store。

8.4 跨文档 vs 同文档

ReferenceEditor 区分两种"未解析"：

场景 kind 提示
同文档内 path 写错 reference "path not found in current document"
跨文档未加载到 project cross-ref "path not in any loaded document"
同文档但 multiplicity 冲突 multiplicity "target container count out of range"

BSWMD 模块管理BSWMD Module Management

BSWMD（Basic Software Module Description）是 schema-side 的 AUTOSAR XML， 描述每个 BSW 模块的合法参数定义。v1.1.0 起，claude-AutosarCfg 支持 从 BSWMD 模板一键生成 ECUC 容器实例（Skeleton），含默认值与类型约束。

9.1 核心概念

术语 含义
Schema-side BSWMD：定义 Com 有什么参数、取值范围、multiplicity。
Value-side ECUC ARXML：具体工程中实际的参数值。
Skeleton 从 BSWMD 生成的空 ECUC 模板（参数取默认值）。
Module BSWMD 顶层 ECUC-MODULE-DEF，如 Com / Det / PduR。
Container BSWMD ECUC-PARAM-CONF-CONTAINER-DEF，可被实例化。

9.2 添加 ECUC 模块的 3 种入口

  • ProjectPanel 芯片行：点击 + Add 直接唤起 picker，预选对应 BSWMD。
  • AppHeader 菜单：Project → New ECUC Module from BSWMD… 不带预选。
  • 右键树根：从 AR-PACKAGE 右键菜单添加（Roadmap）。

9.3 ModuleFromBswmdPicker 流程

9.4 Add / Remove 语义（v1.1.1）

Picker 的"确认"是set-semantic的：

  1. 收集现有 ECUC 实例 从 state.documents 列出 sourceBswmdPath + moduleShortName。
  2. diff 现有 vs 新勾选 算 toAdd = picks - existing，toRemove = existing - picks。
  3. Add 路径 调 useCreateEcucFromBswmd hook，写 ARXML + 注册到 store。

  5. 错误回滚 save 失败第一个就 break，partial 失败单独 toast。
!
Remove 的 dirty-guard：删除 ECUC 文件前若文档 dirty，弹 ConfirmDialog 三选项（Save / Discard / Cancel）。Sprint 16c #3 修复了 "save 失败时仍 delete" 的数据丢失 race。

9.X v1.14+ 增量:模块头解析 (moduleHeader / STD-INCLUDES)

自 v1.14.1 PATCH-G 起,claude-AutosarCfg 解析 BSWMD 时会额外捕获两类关键字段, 用于后续 BSW 代码生成器(v1.11+)构造 cross-module #include 链。

9.X.1 <HEADER> → moduleHeader

每个 BSWMD 顶层 MODULE 节点允许声明一个模块头文件路径,例如:

<MODULE-DESC>
  <SHORT-NAME>EcuC</SHORT-NAME>
  <HEADER>EcuC/EcuC.h</HEADER>
  ...
</MODULE-DESC>

解析后落地为 BswmdModuleDefLite.moduleHeader。若 BSWMD 未提供 <HEADER>,resolveModuleHeader 走 fallback 路径 (moduleShortName + ".h"),但 fallback 也会经过 SEC3 白名单校验 (v1.14.3 R-2.1 pre-ship fix)。

9.X.2 <STD-INCLUDES> → includes[]

BSWMD 可声明模块依赖的标准头(v1.14.2 PATCH-H 接入 emit,实现为 buildSelfIncludes + caller-owned Set dedup): 

<STD-INCLUDES>
  <STD-INCLUDE>Std_Types.h</STD-INCLUDE>
  <STD-INCLUDE>Os.h</STD-INCLUDE>
</STD-INCLUDES>

9.X.3 安全白名单 (SEC3) 与诊断码

所有 moduleHeader 与 includes[] 项必须满足白名单 ^[A-Za-z0-9_./-]+$:不允许 ..、绝对路径、 shell meta($/`/;/& 等)。 校验失败的项触发以下诊断:

Code 触发条件
BSW-SEC-002 include / moduleHeader 路径违反白名单
BSW-SEC-003 include 字符串为空(v1.14.2 H1 wire-up)
BSW-SEC-004 引用目标的 BSWMD 已加载但 moduleHeader 未定义
i
为什么 fallback 也走 SEC3? v1.14.3 R-2.1 之前 resolveModuleHeader 直接用 def.shortName 拼路径, 完全绕开 validateHeaderPath。R-2.1 把 fallback 也纳入 SEC3, 关闭了 Security H1(攻击者构造奇怪 moduleShortName 即可绕过白名单)。

多文件 ECUC 导入Resolve Conflicts Wizard · v1.2.0

v1.2.0 起，Import 菜单引入 EB tresos 风格的 Resolve Conflicts wizard： 把多份 ECUC ARXML 按 module 维度聚合导入到当前 project，撞名时弹三栏 diff 表让用户决策， 最后 atomic commit + 单步撤销。

10.1 设计理念

  • Lazy Merged View：源文档在 commit 前完全不被修改，所有合并是叠加态。
  • Module 维度：以 MODULE-CONFIGURATION.shortName 为单位聚合，跨多份文件。
  • 冲突暴露：同 module 撞名时按 container 维度列出，让用户决定每个冲突。
  • 原子性：commit 时先 snapshot + immutable apply，任一 patch 抛错立即 rollback。
  • 可撤销：单步 undoLastCommit 回到 commit 前状态。

10.2 完整流程

10.3 ModuleSelectionPanel

10.4 DiffTable — 三栏冲突解决

10.5 冲突决策类型（8 种 ImportError / 4 种 Patch）

决策 语义 patch.ts 行为
keep 保留 existing 的值 noop patch
replace 用 incoming 覆盖 setParam patch
merge 字段级合并 multi-patch per-field
add 新增 container addContainer patch
skip 跳过 noop patch
rename 改名 (Sprint 15+) setShortName patch

10.6 原子提交与回滚

Commit 阶段 3 步：

  1. Snapshot 记录 sourceFilesTouched 的完整状态。
  2. Immutable Apply 按 patch 顺序应用，每个 patch 用 Object.freeze 包裹。 任何 patch 抛错 → 跳到 rollback。
  3. Set 状态 仅当全部成功时 set() 提交；importSession 保留供 retry。

回滚（undoLastCommit）：

// store action
const undoLastCommit = () => {
  if (snapshot === null) return;
  set((s) => ({
    ...s,
    documents: restoreFromSnapshot(snapshot),
    importSession: snapshot.importSession,
    viewMode: 'import-merged',
  }));
  toast('已撤销最近一次 commit');
};
!
ViewMode 三态 guard：viewMode ∈ {'single', 'combined', 'import-merged'}。 commit 进行中（commitInFlight === true）时 store 拒绝手动切到其它 mode， 防止半状态被其他组件读取。

常见问题FAQ · Troubleshooting

Q1 · 启动 dev 报 "Please run pnpm build first"

Vite 不会在 dev 模式下服务 main / preload 进程。新 clone 出的仓库必须先 pnpm build 一次产出 dist/main/index.js。 之后 pnpm dev 即可，Vite 会热重载 renderer。

Q2 · Save 后再 reload，发现改动丢失

这是 Sprint 16 修复的 regression（commit f7b69a3）。 早期版本 parser 没有把 DEFINITION-REF 写到 in-memory ParamValue， reload 后丢失；现在所有 5 层（parser / addParameter / addReference / serializer / skeleton）都 stamp definitionRef。请确保版本 ≥ v1.1.1。

Q3 · 1336 cross-ref baseline 是 bug 吗？

不是。S32K148 fixture 是 切片（slices），不构成自包含的 project。 大部分 cross-ref 指向未加载的模块。生产 BSW 工程应趋向 0。详见 CHANGELOG.md v0.8.0 的 Deviations 章节。

Q4 · 跨项目导入支持吗？

v1.2.0 不支持。导入目标必须是 当前 project。跨项目导入 列入 Sprint 15+ Out of Scope 列表。

Q5 · 编辑时校验慢/卡顿

校验是同步的，但 validateProject 在 5 fixture 9.2 MB 上 < 50ms。 如果卡顿，99% 是 IPC 慢（文件 IO）。检查：

  • 是否启用了 Windows Defender 实时扫描 ARXML 目录？
  • 是否把项目放在网络盘 / OneDrive 同步目录？
  • 是否 BSWMD 文件过大（> 50 MB）？v1.0.0+ 已经支持 32 MiB cap。

Q6 · 如何升级而不丢失数据？

manifest.json v1.1.0 时代的绝对路径会被 v1.1.1+ 自动 migrateManifestPaths()， 用户无感。手动操作：升级前 Save All，确保 0 dirty，再覆盖安装。

Q7 · Script Engine 什么时候能上？

Script Engine v1.3.0 spec 已锁定（commit f77acf5），基于 CodeMirror 6 + node:vm sandbox。具体上线节点请关注仓库根 CHANGELOG.md 与 RoadMap.md；详细设计见 docs/superpowers/specs/2026-06-18-script-engine-design.md。

Q8 · 是否支持 Classic 以外的平台？

目前仅支持 AUTOSAR Classic Platform。Adaptive Platform 的 manifest.json、服务接口（SOME/IP、IPC）解析不在 scope。 关注 RoadMap 留意后续动态。

Q9 · BSWMD picker 选过之后又改怎么重置？

picker 是 stateless modal，每次打开都从当前 store 重新解析源 （v1.1.2 stale-seed 修复，82ca016），不会用缓存。 直接关掉重开即可。

Q10 · 怎么报告 bug？

在 GitHub Issues 创建 issue，附：

  • pnpm dev 完整启动日志（截最后 30 行）
  • 复现 ARXML / BSWMD 路径（脱敏后）
  • pnpm verify 的 6 阶段结果
  • Node / pnpm 版本（node -v && pnpm -v）

autosarcfg (CLI) 无头命令行入口(bin/autosarcfg.mjs,pnpm autosarcfg)。 4 个子命令:read · mutate · validate · generate。

B.2 Layer Separation 规则

层 允许依赖 禁止 ESLint 强制
core/ 无（纯 TS） react / react-dom / electron / DOM ✓ no-restricted-imports
shared/ 无（纯 TS types） react / react-dom / electron ✓
main/ electron / node react ✓
preload/ electron react ✓
renderer/ react / zustand electron（必须走 preload） ✓

B.4 IPC 通道速查

通道 方向 作用
arxml:open R → M 原生对话框 + 选文件
arxml:parse R → M 解析 + 返回 documents
arxml:save R → M 序列化单文件到磁盘
project:new R → M 创建 manifest + 模板文件
project:open R → M 读取 manifest 装载 documents
project:close R → M 关闭 + 清空 store
bswmd:read R → M 解析 BSWMD，返回 schema
bswmd:parse R → M BSWMD 解析（带 cap 32 MiB）
templates:list R → M 列出内置模板（opt-in）
templates:copy R → M 复制模板到目标目录
pick:dir R → M 原生目录选择对话框
pick:file R → M 原生文件选择对话框
project:writeArxmlBatch R → M 原子批量写（multi-ECUC Save All）
project:deleteArxml R → M 删除 ECUC 文件（v1.1.0+）

B.5 性能基线

  • 打开 5 fixture (9.2 MB): ~ 50 ms parse + 30 ms validate
  • 单参数编辑触发校验: < 5 ms（同步）
  • debounced 校验: 300 ms 后兜底
  • Save 单文件: < 20 ms（writeFile）
  • Save All 5 文件: ~ 80 ms（顺序写）
  • Cross-ref 索引构建: ~ 12 ms（14200 条目）
