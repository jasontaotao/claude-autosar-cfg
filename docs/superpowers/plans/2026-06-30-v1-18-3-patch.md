### 2.1 Edit: `src/core/generator/post-process.ts`

Replace the `try` block at lines 71-86 with the fsync-aware version:

```ts
try {
  await mkdir(parentReal, { recursive: true });
  const tmpPath = `${absPath}.${process.pid}.${Date.now()}.tmp`;
  await writeFile(tmpPath, content, 'utf8');
  // v1.18.3 PATCH — fsync the temp before rename so generator
  // output survives a crash. Mirrors the main-side `writeAtomic`
  // pattern at `src/main/io/writeAtomic.ts:36-41`:
  //   open a second time in r+ (writeFile alone does not surface
  //   its descriptor), sync the data + metadata, close the handle,
  //   THEN rename.
  // Without fsync, `writeFile` may leave the data in the OS page
  // cache; `rename` makes the inode swap visible but the content
  // can still be lost on power loss / OS crash.
  const fh = await open(tmpPath, 'r+');
  try {
    await fh.sync();
  } finally {
    await fh.close();
  }
  await rename(tmpPath, absPath);
} catch (e) {
  // ... existing diagnostic + continue ...
}
```

Plus update the import at line 1:

## 5. Risk Callouts

- **`open` instead of `open` then `write`**: keeps the existing `writeFile` write path (no rewrite of UTF-8 encoding options). Mirrors main-side `writeAtomic` exactly.
- **`r+` mode for fsync**: required because `fsync` on an O_WRONLY handle is implementation-defined on some platforms. `r+` is the canonical "I have a handle to this file, sync its data" mode.
- **`try/finally` for `close()`**: if `sync()` throws, the handle must close before the error propagates — otherwise we leak file descriptors under load.
- **No behavior change for happy path**: existing tests pass without modification; the only observable difference is that fsync adds a disk-sync barrier between writeFile and rename.
- **Performance note**: fsync adds ~1-10ms per file (SSD/HDD-dependent). For the generator pipeline that writes dozens-to-hundreds of files, this is a non-trivial cost. Acceptable for crash-safety; if performance becomes a concern, can add a feature flag in a future PATCH.
