## 2. Architecture

### 2.1 New file: `src/main/headless/push-emitters.ts`

```ts
// v1.18.1 PATCH — Headless push channel emitters.
//
// Two emitters that ship the `HEADLESS_MUTATE_APPLIED` and
// `HEADLESS_VALIDATE_RESULT` events to the renderer via
// `webContents.send`. Mirror the `SCRIPT_PROGRESS` push pattern
// in `src/main/ipc/script-handler.ts:312-336`:
//
//   1. Snapshot `getMainWindow()` (window may close mid-emit).
//   2. Guard with `null === null || mainWindow.isDestroyed()`.
//   3. Send via `mainWindow.webContents.send(channel, payload)`.
//
// In CLI mode (standalone `bin/autosarcfg.mjs`), there is no main
// window — `getMainWindow()` returns null and the emitters are
// no-ops. This is correct: the CLI does not own a renderer; the
// push channel is exclusively for the future GUI bridge (per
// v1.17.0 spec §15.1 deferral note).

import { getMainWindow } from '../window.js';
import {
  HEADLESS_MUTATE_APPLIED,
  HEADLESS_VALIDATE_RESULT,
} from '../../shared/headless/ipc-contract.js';
import type {
  MutateAppliedEvent,
  ValidateResultEvent,
} from '../../shared/headless/ipc-contract.js';

/**
 * Emit a `HEADLESS_MUTATE_APPLIED` event to the renderer.
 * No-op if no main window (CLI mode) or window is destroyed.
 */
export function emitMutateApplied(event: MutateAppliedEvent): void {
  const mainWindow = getMainWindow();
  if (mainWindow === null || mainWindow.isDestroyed()) return;
  mainWindow.webContents.send(HEADLESS_MUTATE_APPLIED, event);
}

/**
 * Emit a `HEADLESS_VALIDATE_RESULT` event to the renderer.
 * No-op if no main window (CLI mode) or window is destroyed.
 */
export function emitValidateResult(event: ValidateResultEvent): void {
  const mainWindow = getMainWindow();
  if (mainWindow === null || mainWindow.isDestroyed()) return;
  mainWindow.webContents.send(HEADLESS_VALIDATE_RESULT, event);
}
```

### 2.2 Edit: `src/shared/headless/ipc-contract.ts`

Add two wire types:

```ts
/**
 * Wire shape for `HEADLESS_MUTATE_APPLIED` push event (v1.18.1 PATCH).
 * Subset of `MutateResult` sufficient for renderer to refresh project
 * tree + dirty flag. The full `MutateResult` envelope is too large
 * for a push event (includes `durationMs`, `dryRunPreview`, etc.).
 */
export interface MutateAppliedEvent {
  readonly patchId: string;
  readonly applied: number;
  readonly warnings: ReadonlyArray<MutationStepWarning>;
}

/**
 * Wire shape for `HEADLESS_VALIDATE_RESULT` push event (v1.18.1 PATCH).
 * Wraps the canonical `ValidateResult` envelope; renderer can subscribe
 * for live validation panels without polling.
 */
export type ValidateResultEvent = ValidateResult;
```

### 2.3 Edit: `src/main/ipc/headless-stubs.ts`

Call the emitters from the stub so they're exercised in tests. The stub currently returns `{ ok: true, stub: true }` — no real mutate happens, so no `MutateAppliedEvent` is emitted (correct: nothing happened). But `headlessRunCommandStub` should accept a `HeadlessRunRequest` with a real command + forward to the CLI dispatcher when the stub is replaced; for v1.18.1, leave the stub return shape but export `emitMutateApplied` + `emitValidateResult` for future use.

**Decision**: do NOT wire the emitters into the stub in this PATCH. The stub's contract is "real bridge not wired yet"; calling emitters from a stub would imply work was done. **Wire only into a new exporter hook** (the emitters themselves). Future PATCH (when stub is replaced) calls the emitters from the real dispatcher.

## 5. Risk Callouts

- **Window-null branch is critical**: when CLI runs standalone, `getMainWindow()` returns null. Emitters MUST be no-ops in that case (not throw). The `if (mainWindow === null || mainWindow.isDestroyed()) return` guard handles both paths.
- **Window-destroyed race**: `setMainWindow(null)` from `registerMainWindowCloseHandler` (v1.17.1) clears the accessor BEFORE the destroyed event fires on every code path, so `mainWindow === null` is the dominant case. The `isDestroyed()` guard is defense-in-depth for the race between "user closes window" and "main process emits event from a long-running operation".
- **No `import` of electron types in test file**: the test mocks `webContents.send` via duck-typed fake (same as `script-progress-emit.test.ts:31-34`). No `electron` import.
