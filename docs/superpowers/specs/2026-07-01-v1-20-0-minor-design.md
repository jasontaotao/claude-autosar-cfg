## 3. `isPathInsideReal` (Symlink defense, Batch 2)

### 3.1 New helper: `src/shared/paths/isPathInsideReal.ts`

```ts
// shared/paths/isPathInsideReal.ts
// Async realpath-based path containment check. Defense-in-depth
// against attacker-controlled symlinks inside the project dir:
//   - manifest entry like `bswmdPaths: ["link-to-/etc/passwd"]`
//   - where `link-to-/etc/passwd` is a real symlink in the project dir
//   - pure-string `isPathInside` returns true (string path stays inside)
//   - realpath follows the link → `/etc/passwd` → returns false
//
// Used by IPC handlers in `src/main/ipc/` that gate security-sensitive
// reads/writes.

import { realpath } from 'node:fs/promises';
import { isAbsolute, relative, sep } from 'node:path';
import { isPathInside } from './isPathInside.js'; // fallback for non-existent paths

/**
 * Returns true iff `child` is strictly inside `parent` after resolving
 * all symlinks via `realpath`. Throws nothing — falls back to the pure-
 * string `isPathInside` if `realpath` fails on a non-existent path.
 */
export async function isPathInsideReal(child: string, parent: string): Promise<boolean> {
  let childReal: string;
  let parentReal: string;
  try {
    childReal = await realpath(child);
  } catch {
    childReal = child; // fall through to string compare
  }
  try {
    parentReal = await realpath(parent);
  } catch {
    parentReal = parent;
  }
  // Delegate to the existing sync helper, which already handles
  // Windows case-insensitivity, UNC, ..-traversal, etc.
  return isPathInside(childReal, parentReal);
}
```

## 5. GUI `applyMutation` Parity (C2.4, Batch 4)

### 5.1 Current state

`src/renderer/store/useScriptStore.ts:313` `applyMutation()`:

- Reads `result.mutations[]` (script-shaped: `{ kind: 'set-param' | 'add-child' | 'remove-child', ... }`).
- Switch-cases each `m.kind`, calls `useArxmlStore.updateParam` / `addContainer` / `deleteContainer` directly.
- Captures any `error` / `pendingDelete` from the store.
- Serializes via `serializeArxml` and IPC-saves via `window.autosarApi.projectSave`.

### 5.2 New flow (parity with CLI)

```ts
applyMutation: async (): Promise<void> => {
  const result = get().runResult;
  if (result === null) return;

  const { useArxmlStore } = await import('./useArxmlStore.js');
  const { applyPatchSteps } = await import('../../core/mutation/applyPatchSteps.js');
  const { serializeArxml } = await import('../../core/arxml/serializer.js');

  const arxmlState = useArxmlStore.getState();
  const filePath = arxmlState.activeDocumentPath;

  // No active doc → nothing to apply. Preserve stub contract.
  if (arxmlState.doc === null || filePath === null) {
    set({ runResult: { ...result, mutations: [] }, dirty: false });
    return;
  }

  // Build PatchDocument from script-shaped mutations.
  const patchDoc = {
    autosarcfgPatchVersion: '1' as const,
    steps: result.mutations.map(scriptMutationToPatchStep),
  };

  // Apply via the same engine the CLI uses.
  const applyResult = applyPatchSteps(arxmlState.doc, patchDoc.steps);

  // Thread warnings back into runResult.warnings.
  const warnings = applyResult.warnings.map((w) => ({
    stepIndex: w.stepIndex,
    kind: w.kind,
    message: w.message,
  }));

  if (applyResult.errors.length > 0) {
    set({
      runResult: {
        ...result,
        mutations: [],
        warnings,
        errorMessage: applyResult.errors[0].message,
        runState: 'failed',
      },
      dirty: false,
    });
    return;
  }

  // Update store doc (for in-memory display).
  useArxmlStore.setState({ doc: applyResult.doc });

  // Persist via project:save IPC.
  const serialized = serializeArxml(applyResult.doc);
  if (!serialized.ok) {
    set({
      runResult: {
        ...result,
        mutations: [],
        warnings,
        errorMessage: serialized.error.message,
        runState: 'failed',
      },
      dirty: false,
    });
    return;
  }

  await window.autosarApi.projectSave({
    manifestPath: arxmlState.projectPath,
    documents: [{ path: filePath, content: serialized.value }],
  });

  set({
    runResult: {
      ...result,
      mutations: [],
      warnings,
      runState: 'applied',
    },
    dirty: false,
  });
};
```

### 5.3 Mapping function: `scriptMutationToPatchStep`

NEW file `src/renderer/store/helpers/scriptMutationToPatchStep.ts`:

```ts
import type { PatchStep } from '../../../core/mutation/applyPatchSteps.js';

type ScriptMutation =
  | { kind: 'set-param'; containerPath: string; paramName: string; newValue: unknown }
  | { kind: 'add-child'; containerPath: string; newShortName: string }
  | { kind: 'remove-child'; containerPath: string };

export function scriptMutationToPatchStep(m: ScriptMutation): PatchStep {
  switch (m.kind) {
    case 'set-param':
      return {
        op: 'set-param',
        containerPath: m.containerPath,
        paramName: m.paramName,
        value: m.newValue as string | number | boolean | null,
      };
    case 'add-child':
      return {
        op: 'add-child',
        parentPath: m.containerPath,
        shortName: m.newShortName,
      };
    case 'remove-child':
      return {
        op: 'remove-with-cascade',
        containerPath: m.containerPath,
        cascade: false, // existing script UX is non-cascade; cascade dialog preserved
      };
  }
}
```

### 5.4 Existing behavior preservation

- **Cascade dialog for `remove-child`**: today, `useArxmlStore.deleteContainer` opens `pendingDelete` dialog when references exist; the doc is NOT mutated. The new path uses `applyPatchSteps` which calls into the same `deleteContainer` logic for cascade detection. **Risk**: `applyPatchSteps` may not currently expose cascade-dialog semantics; verify during implementation. If not, fall back to the existing direct-call path for `remove-child` (only `set-param` + `add-child` go through `applyPatchSteps`). This is a TDD-driven decision; spec keeps the option open.
- **`writeErrors` capture**: existing test `useScriptStore.test.ts:369` asserts `set-param: path not found` is surfaced as a mutation error. The new path's `applyResult.errors[]` carries this — preserved.
- **`pendingDelete` capture**: existing test line 704 asserts cascade short-circuit is surfaced. The new path needs to detect the same short-circuit (which `applyPatchSteps` may or may not surface today — needs Phase 0 follow-up).
