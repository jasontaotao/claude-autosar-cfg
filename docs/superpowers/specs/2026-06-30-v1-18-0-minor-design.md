## 5. Renderer Crash Recovery (PB-1 + PB-4, Batch 2)

### 5.1 Main-side webContents hooks (PB-1)

Add to `createMainWindow` in `src/main/index.ts` after `registerMainWindowCloseHandler(mainWindow)` (v1.17.1 PATCH wiring site):

```ts
// PB-1 — render-process crash recovery.
mainWindow.webContents.on('render-process-gone', (_e, details) => {
  logFatal('render-process-gone', details);
  dialog
    .showMessageBox(mainWindow, {
      type: 'error',
      title: 'Renderer crashed',
      message: `Renderer process exited unexpectedly (reason: ${details.reason}).`,
      buttons: ['Reload', 'Quit'],
      defaultId: 0,
    })
    .then(({ response }) => {
      if (response === 0) mainWindow.webContents.reload();
      else app.quit();
    });
});

mainWindow.webContents.on('unresponsive', () => {
  logFatal('webContents-unresponsive', new Error('Renderer unresponsive > 30s'));
  dialog
    .showMessageBox(mainWindow, {
      type: 'warning',
      title: 'Renderer unresponsive',
      message: 'The renderer is unresponsive. Continue waiting?',
      buttons: ['Wait', 'Reload'],
      defaultId: 0,
    })
    .then(({ response }) => {
      if (response === 1) mainWindow.webContents.reload();
    });
});

mainWindow.webContents.on('gpu-process-crashed', (_e, killed) => {
  logFatal('gpu-process-crashed', new Error(`GPU process crashed (killed=${killed})`));
  mainWindow.webContents.reload();
});
```

### 11.2 C13 re-plan rationale (deferred from v1.18.0 implementation)

Per Phase 0 follow-up during T4 dispatch (2026-06-30), 4 critical drifts were discovered:

1. **AppHeader.tsx is NOT 6 independent sub-components** — it is one cohesive function component with module-level helpers (`formatParseError`, `saveAllDirty`). The aspirational spec §2.1 invented `ActionMenu`, `ToolbarButtons`, `StatusBar`, `WindowControls`, `ProjectBreadcrumb` as sub-components — they do not exist in the file. Inventing them would require rewriting body content (not just moving code), violating the "zero behavior change" invariant.

2. **useProjectActions.ts has a `saveProject` declaration-order closure** — `saveProject` (line 292) is declared first so that 5 sibling `useCallback`s can close over it via `guardedDirtySwitch`. Splitting into 7 sub-hooks breaks this closure unless the public API changes (callers pass `saveProject` as an arg, breaking signature).

3. **`AppHeader.scripts.test.tsx` exists at `src/renderer/components/__tests__/`** — it asserts on the monolithic DOM (e.g. `data-testid="btn-scripts-toggle"`). The split plan §2.1 ignored this test; refactor risk was not flagged.

4. **T3 (IPC-4) already created `src/renderer/hooks/useProjectActions/__tests__/useProjectOpen.saveProject.test.tsx`** (217 lines) — this file imports from `'../../useProjectActions.js'` (assumes subdir barrel structure). Plan §2.2 listed this test as NEW under T4, but it was actually created by T3. Conflict on commit.

**Decision**: defer C13 to v1.18.4 PATCH with proper re-planning. Honest scope options:

- **Option B** (preferred for v1.18.4 PATCH brief): co-locate module-level helpers + types into separate files, keep the function bodies monolithic. Yields 820 → ~720 main + 100 helper lines; 791 → ~660 main + 130 helper lines. Satisfies "many small files" without inventing components.
- **Option A** (literal execution of original spec): invent 6 AppHeader sub-components + 7 useProjectActions sub-hooks. ~40% of work is rewriting body content. Public API change required for useProjectActions. Test consumer updates required. **Net value low, risk high** — same risk profile as the v1.17.0 aspirational-spec BLOCK.

**Rationale for NOT executing in v1.18.0**: C13 is pure refactor with no behavior change. T1, T2, T3 (shipped) + T5-T8 (PB-1/3/4 + C8, user-facing) deliver more MINOR value per commit. C13 deserves its own dedicated PATCH cycle with re-planning.

**Test count impact**: v1.18.0 ships 7 items instead of 8; expected test delta +34 → +22 (subtract C13's planned 12 tests: 5 AppHeader + 7 useProjectActions sub-component smoke).
