3. 运行一次脚本的时序

1
用户 → Renderer
在 Scripts 面板点 ▶ Run（或在编辑器里按 Ctrl+Enter）
2
Renderer → Main (IPC script:run)
发送 { projectId, id, timeoutMs?: 5000 }。如果当前已有运行，UI 显示 正在执行… 状态并 disable 按钮
3
Main: scriptHandler
从 manifest 读出脚本源 + 依赖项列表；拍快照（structural clone）项目状态到 WorkingCopy
4
Main: importResolver
扫描脚本里的 import，按项目 manifest 解析依赖；DAG 拓扑排序；循环依赖直接报错退出
5
Main: vmRunner
创建 vm.createContext(ctx)（白名单 API）→ vm.Script(rewritten).runInContext() → 启动 timeout watchdog
6
Sandbox 内部（用户脚本）
脚本调 ctx.project.findContainers(...)、ctx.container.getParam(...).setValue(...)、ctx.log.info(...)、ctx.validator.addViolation(...)
7
Main → Renderer (IPC script:progress 事件流)
每条 ctx.log 实时推回；校验项累积；UI 输出面板追加
8
Main: 完成 / 失败 / 超时
成功：把 WorkingCopy 的 mutations 跟原项目做 diff，回传 { mutations, violations, logs }；renderer 让用户选择 commit / discard
失败：完整 stack trace（去混淆回原始源码行号）+ 丢弃 mutations + 错误高亮到编辑器对应行
超时：sandbox 不可中断，标记 timedOut，等自然结束（V0.1）
9
Renderer
校验项 kind: 'script:*' 进入主 Validation 面板（与 7 种原生错误并列、独立色块）
commit 后的 mutations 应用到 zustand store，触发既有 6 种原生校验

4. ctx API 形状

所有 API 只读 core 层类型；mutation 走 WorkingCopy 缓冲，永不直接改原项目。Zod schema 校验外部输入。

ctx.project Project
项目根对象。所有遍历从这里开始。
.documents : Document[] .findContainers({type, def, predicate}) .getContainer(path) .buildPathIndex() => Map
ctx.document Document
单个 .arxml 文档（可读写 WorkingCopy）
.path : string .findContainers(...) .containers : Container[] .save() : 标记待 commit
ctx.container Container
包装层，方便链式调用。也可由 findContainers() 返回值直接调。
.path : string .def : string .params : Param[] .children : Container[] .addChild(name)
ctx.param Param
单参数读写入口。返回值自带类型方法。
.name, .type, .value .asInteger() / asString() / asBoolean() .setValue(v) → 走 Zod 校验
ctx.validator ValidatorSink
脚本专属校验项出口，会进入主 Validation 面板。
.addViolation({   kind: 'script:*',   severity,   containerPath,   message,   data? })
ctx.schema BswmdSchema
读 BSWMD 定义（已加载到项目的所有 schema 文件）。
.getDef(path) : ParamDef | ContainerDef .getAllParamDefs(modulePath)
ctx.log Logger
推回 renderer 输出面板的实时日志。
.info(msg) / .warn(msg) / .error(msg) .debug(msg) — V0.1 默认隐藏
ctx.utils Utils
与项目模型解耦的小工具。
.path : 与 core/arxml/path 同构 .now() : ISO string .assert(cond, msg)

7. 待你拍板的开放问题

Q1 · 事务语义（必答）
脚本对项目的修改走"快照 + commit/discard"还是"直接改原项目"？前者安全（失败可回滚）但 UI 多一步；后者简单但失败可能半改半留。我推荐前者。
Q2 · Output 面板要不要嵌主窗口
是把 Scripts 面板做成主窗口的可隐藏侧栏（贴近 ASCET 体验），还是做一个独立 BrowserWindow / Modal？前者更紧凑，后者能做得更"IDE 风"。
Q3 · 脚本 kind 分类（validator/transformer/report）
要不要在 UI 上做这种分类（决定侧栏小图标和默认按钮文案）？还是统一 "Script" 一种 kind？
Q4 · 范围外但相关
要不要同时把 "headless CLI 模式" 作为下个 sprint 路线图列出来？本设计只做 GUI 内嵌引擎；CLI 是另一次设计。
