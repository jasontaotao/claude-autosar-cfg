### I-1 ‚Äî Remove misleading `as 'integer'` cast on unknown-kind test

**Current** (`src/core/generator/__tests__/c-type-for-kind.test.ts:71-77`):

```ts
it('unknown kind (both modules) ‚Üí uint8 (per-module fail-safe)', () => {
  // Cast needed because the production type is `EcuCParamDefLike |
  // McuParamDefLike` (strict union); the fail-safe branch must
  // accept any string for defense-in-depth, but the test exercises
  // that contract without bypassing the type system at the call
  // sites that use this helper legitimately.
  expect(cTypeForKind({ kind: 'unknown-kind' as 'integer' }, 'EcuC')).toBe('uint8');
  expect(cTypeForKind({ kind: 'unknown-kind' as 'integer' }, 'Mcu')).toBe('uint8');
});
```

**Problem**: the `as 'integer'` cast tells the reader the test is exercising the `integer` arm when the runtime value `'unknown-kind'` actually exercises the fail-safe backstop (after falling through `cTypeForBasicKind` ‚Äî which has 4 named cases ‚Äî and hitting `cTypeForKind`'s default-catch). The cast only exists because the production parameter type is `EcuCParamDefLike | McuParamDefLike` (strict union of two narrow types).

**Fix** (Option A ‚Äî type widening at the test site, RECOMMENDED):

Widen the test-local helper signature via a TypeScript intersection cast that preserves the test's intent without misleading the reader:

```ts
it('unknown kind (both modules) ‚Üí uint8 (per-module fail-safe)', () => {
  // The fail-safe branch must accept any string for defense-in-depth.
  // We use a `satisfies` cast that keeps the production type narrow at
  // the call sites but widens to `{ kind: string }` here so the unknown
  // kind is exercised without an `as 'integer'` lie.
  const unknownDef = { kind: 'unknown-kind' } satisfies { kind: string };
  expect(cTypeForKind(unknownDef as unknown as Parameters<typeof cTypeForKind>[0], 'EcuC')).toBe(
    'uint8',
  );
  expect(cTypeForKind(unknownDef as unknown as Parameters<typeof cTypeForKind>[0], 'Mcu')).toBe(
    'uint8',
  );
});
```

The `as unknown as Parameters<typeof cTypeForKind>[0]` is a double-cast that preserves the type-system intent ("the production signature is narrow, this is an exception") while making the cast path obvious (`Parameters<...>[0]` is the signature's first param type, not a misleading literal type).

**Alternative** (Option B ‚Äî split into 2 tests, defer to v1.15.4):

Keep the cast but split the test into "known-kind arm reaches fail-safe" (uses real union member) + "literal string unknown-kind bypasses type system" (uses `as never`). More verbose; deferred.

**Decision**: Option A.

**Test impact**: same 1 test case renamed/refactored, no count change.

### M-T4-1 ‚Äî Chain M-2.1 assertions with `.not.toContainEqual(...)`

**Current** (`src/core/generator/__tests__/pipeline.test.ts:447-457`):

```ts
// v1.15.2 PATCH (M-2.1) ‚Äî tightened from v1.15.1 M2.1's
// BSW-SEC-004-only assertion. The v1.15.1 version did not
// guard against other stage-1 diagnostics (BSW-SEC-002,
// BSW-SEC-003, ECUC-GEN-001) firing silently. The tightened
// version forbids any stage-1 ERROR and bounds WARNING at
// <= 1 (BSW-SEC-003 known-warn tolerance per v1.14.2 H1;
// future tighten when BSW-SEC-003 is fully silenced).
expect(result.diagnostics.filter((d) => d.severity === DiagnosticSeverity.ERROR)).toHaveLength(0);
expect(
  result.diagnostics.filter((d) => d.severity === DiagnosticSeverity.WARNING).length,
).toBeLessThanOrEqual(1);
// BSW-SEC-004 specifically is silent (ref target has moduleHeader).
expect(
  result.diagnostics.some((d) => d.code === DiagnosticCode.BSW_SEC_MISSING_TARGET_HEADER),
).toBe(false);
// Stage 2 ran (SpyGen.emit was called for EcuC).
expect(result.artifacts.size).toBeGreaterThan(0);
expect(result.exitCode).toBe(0);
```

**Problem**: the `expect(result.diagnostics.some(...))` style is verbose and inconsistent with vitest's preferred `expect(arr).not.toContainEqual(partial)` for "this kind of object is not in the array" checks.

**Fix** (chain the diagnostics checks):

```ts
// v1.15.2 PATCH (M-2.1) + v1.15.3 PATCH (M-T4-1) ‚Äî tightened from
// v1.15.1 M2.1's BSW-SEC-004-only assertion. v1.15.3 chains the
// negative checks via `.not.toContainEqual(...)` for symmetry with
// the rest of the v1.15.0+ vitest assertion style.
const { diagnostics, artifacts, exitCode } = result;
// No stage-1 ERROR may fire.
expect(diagnostics.filter((d) => d.severity === DiagnosticSeverity.ERROR)).toHaveLength(0);
// BSW-SEC-003 known-warn tolerance per v1.14.2 H1 (‚â§1 WARN).
expect(diagnostics.filter((d) => d.severity === DiagnosticSeverity.WARNING)).toHaveLength(
  lessThanOrEqual(1),
);
// BSW-SEC-004 specifically is silent (ref target has moduleHeader).
expect(diagnostics).not.toContainEqual(
  expect.objectContaining({ code: DiagnosticCode.BSW_SEC_MISSING_TARGET_HEADER }),
);
// Stage 2 ran (SpyGen.emit was called for EcuC).
expect(artifacts.size).toBeGreaterThan(0);
expect(exitCode).toBe(0);
```

Note: the WARN bound check is changed from `.length` + `.toBeLessThanOrEqual(1)` to `.toHaveLength(lessThanOrEqual(1))`. `lessThanOrEqual` is vitest's asymmetric matcher; same semantics, cleaner expression. If asymmetric matcher syntax is undesired (reviewer preference), fall back to:

```ts
expect(
  diagnostics.filter((d) => d.severity === DiagnosticSeverity.WARNING).length,
).toBeLessThanOrEqual(1);
```

**Test impact**: same 1 test, assertion style change. No count change. Coverage unchanged (96.01% / 86.98% / 95.58%).
