## Architecture

### M1 — `validateRefTargetHeaders` signature migration

**Current state** (post-v1.15.0):

```ts
export function validateRefTargetHeaders(
  bswmdIndex: ReadonlyMap<string, BswmdIndexForModuleHeaderPaths>,
  ecucValues: ReadonlyMap<string, EcucModuleConfigurationValuesInput>,
): readonly Diagnostic[] {
  for (const [moduleShortName, values] of ecucValues) {
    for (const ref of values.references ?? []) {
      const targetDef = bswmdIndex.get(ref.targetModule);
      if (targetDef && targetDef.moduleHeader === undefined) {
        /* push BSW-SEC-004 */
      }
    }
  }
  return out;
}
```

Pipeline call site (`pipeline.ts:108-113`):

```ts
diagnostics.push(
  ...validateRefTargetHeaders(
    args.bswmdIndex as BswmdIndexForModuleHeaderPaths,
    args.ecucValues as EcucIndexForRefTargetHeaders,
  ),
);
```

**Asymmetry flagged by B-2 review**: every other Stage-1 validator takes `tree: NormalizedConfigTree` (e.g. `validateReferences(tree)`, `validateMultiplicity(tree)`, `validateTypeMatches(tree)`, `validateRange(tree)`, `validateOrdering(tree)`, `validateUniqueShortNames(tree)`). `validateRefTargetHeaders` is the lone outlier, taking the raw pipeline args.

**Fix**: migrate the signature to take `NormalizedConfigTree` (already built by `normalizeToTree` in Stage 1, line 89). Use `tree.valuesByModule` (instead of `ecucValues`) and `tree.bswmdIndex` (instead of `bswmdIndex`).

```ts
export function validateRefTargetHeaders(tree: NormalizedConfigTree): readonly Diagnostic[] {
  for (const [moduleShortName, values] of tree.valuesByModule) {
    for (const ref of values.references ?? []) {
      const targetDef = tree.bswmdIndex.get(ref.targetModule);
      if (targetDef && targetDef.moduleHeader === undefined) {
        /* push BSW-SEC-004 */
      }
    }
  }
  return out;
}
```

Cleaner alternative: use `tree.references` directly (the collected `ReferenceEdge[]` from `normalizeToTree`):

```ts
for (const ref of tree.references) {
  const targetDef = tree.bswmdIndex.get(ref.targetModule);
  if (targetDef && targetDef.moduleHeader === undefined) {
    /* push */
  }
}
```

This eliminates the per-module iteration AND the `values.references ?? []` defensive check. `tree.references` is the single source of truth.

**Recommended form**: the `tree.references` form. Same diagnostic shape, fewer indirections, no per-module loop. The `moduleShortName` in the pushed diagnostic becomes `ref.sourceModule` (matches `validateReferences`'s style at `emit/reference.ts:53`).

**Pipeline call site** (after M1):

```ts
diagnostics.push(...validateRefTargetHeaders(tree));
```

**Type-alias cleanup** (in `pipeline.ts`):

The two type aliases `BswmdIndexForRefTargetHeaders` and `EcucIndexForRefTargetHeaders` (added in v1.15.0 B-2.3 at `pipeline.ts:73-74`) become dead code. Remove them in the same commit. The `validateRefTargetHeaders` import (added in v1.15.0 B-2.3) is retained.

**Behavior preservation**: identical diagnostic shape (same `code`, `severity`, `moduleShortName`, `ecucPath`, `message`). Same `BSW-SEC-004` push for the same condition. All 2466 + 2 skipped baseline tests pass unchanged.

### M2 — Positive integration test

**Current state**:

The existing B-2.6 integration test "Stage 2 runs normally when no BSW-SEC-004 fires" (pipeline.test.ts:354-388) tests only "no refs at all → no BSW-SEC-004". This is a trivially-true path: `validateRefTargetHeaders` returns `[]` for any `ecucValues` map without `references[]` entries.

The real positive case ("ref target HAS moduleHeader → no push") is asserted only at the unit-test level in `validate-ref-target-headers.test.ts`. An end-to-end regression where Stage-1 misroutes a happy-path ref would slip past `pipeline.test.ts` and only surface in the unit-test layer.

**Fix**: add 1 integration test where:

- bswmdIndex: EcuC with `moduleHeader: 'EcuC/EcuC_Cfg.h'`
- ecucValues: EcuC with `containers: [{shortName: 'Config'}]` (so `validateReferences`'s instance-level check passes — `ref.targetPath` tail `'Config'` must match a real container instance) AND EcuC has a `references: [{path, targetModule: 'Mcu', targetPath: '...'}]` entry — wait, that's the wrong direction. Let me re-think.

**The test setup that works**:

- bswmdIndex: Mcu (shortName, moduleHeader) + EcuC (shortName, moduleHeader)
- ecucValues: Mcu (with `references: [{path, targetModule: 'EcuC', targetPath: 'EcuC/Config'}]`) + EcuC (with `containers: [{shortName: 'Config'}]` for the instance-level check)

The flow:

- `validateReferences`: Mcu's ref to EcuC → EcuC in `valuesByModule` (EcuC ecucValues entry exists) → `targetPath` tail `'Config'` matches EcuC's `containers[0].shortName === 'Config'` → no error
- `validateModuleHeaderPaths`: Mcu has valid `moduleHeader`; EcuC has valid `moduleHeader` → no BSW-SEC-002
- `validateRefTargetHeaders`: Mcu's ref → EcuC in bswmdIndex with `moduleHeader` set → no BSW-SEC-004
- `validateMultiplicity` / others: no containers/params mismatch
- S6 early-break: no
- Stage 2: Mcu has no generator → `ECUC_GEN_NO_GENERATOR` WARN; EcuC has SpyGen → 1 artifact
- Result: BSW-SEC-004 not in diagnostics, Stage 2 runs, artifacts.size > 0, exitCode = 0

This is the genuine "positive path" the B-2 review asked for. The M2 task adds this 1 case to `pipeline.test.ts`.

### B-5 — `cTypeForBasicKind` extraction

**Current state** (post-v1.14.3 C-2 fix):

`EcuCGenerator.cTypeForKind` (`ecuc.ts:147-166`):

```ts
function cTypeForKind(def: EcuCParamDefLike): string {
  switch (def.kind) {
    case 'integer':
      return integerToCType(def.min ?? 0, def.max ?? 0);
    case 'boolean':
      return 'uint8';
    case 'string':
      return 'const char*';
    case 'float':
      return 'float32';
    case 'enumeration':
      return 'uint8';
    case 'reference':
      return `const ${def.targetType ?? 'void'} * const`;
    case 'function-name':
      return def.signature ?? 'void';
    default:
      return 'uint8';
  }
}
```

`McuGenerator.cTypeForKind` (`mcu.ts:124-141`):

```ts
function cTypeForKind(def: McuParamDefLike): string {
  switch (def.kind) {
    case 'integer':
      return 'uint32'; // Mcu clock ref points per AUTOSAR convention
    case 'boolean':
      return 'uint8';
    case 'string':
      return 'const char*';
    case 'float':
      return 'float32';
    case 'enumeration':
      return 'uint8';
    default:
      return 'uint8';
  }
}
```

The 5 common arms (`boolean`, `string`, `float`, `enumeration`, `default`) are identical. Only `integer` differs (EcuC: min/max-aware, Mcu: hardcoded `uint32`). EcuC has 2 extra arms (`reference`, `function-name`).

**Fix**: extract the 5 common arms to a shared helper in `_shared.ts`:

```ts
/**
 * v1.15.1 PATCH (B-5) — shared `cTypeForKind` for the 5 arms that
 * are byte-identical between EcuC and Mcu generators (per v1.14.3
 * C-2 fix that confirmed the default arms are identical). The
 * `integer` arm stays per-module: EcuC uses `integerToCType(min,
 * max)` (min/max-aware), Mcu hardcodes `'uint32'` per AUTOSAR
 * convention for clock reference points. EcuC's `reference` and
 * `function-name` arms also stay per-module (Mcu doesn't model
 * those kinds yet).
 *
 * If B-3 (type-driven refactor) lands, this helper becomes the
 * `default` arm of the unified dispatcher and the per-module
 * `cTypeForKind` functions are deleted.
 */
export function cTypeForBasicKind(kind: string): string {
  switch (kind) {
    case 'boolean':
      return 'uint8';
    case 'string':
      return 'const char*';
    case 'float':
      return 'float32';
    case 'enumeration':
      return 'uint8';
    default:
      return 'uint8';
  }
}
```

EcuC's `cTypeForKind` becomes:

```ts
function cTypeForKind(def: EcuCParamDefLike): string {
  switch (def.kind) {
    case 'integer':
      return integerToCType(def.min ?? 0, def.max ?? 0);
    case 'reference':
      return `const ${def.targetType ?? 'void'} * const`;
    case 'function-name':
      return def.signature ?? 'void';
    default:
      return cTypeForBasicKind(def.kind);
  }
}
```

Mcu's `cTypeForKind` becomes:

```ts
function cTypeForKind(def: McuParamDefLike): string {
  switch (def.kind) {
    case 'integer':
      return 'uint32';
    default:
      return cTypeForBasicKind(def.kind);
  }
}
```

**Behavior preservation**: identical output for all 2454 baseline tests. EcuC `integer` arm unchanged (still uses `integerToCType`). Mcu `integer` arm unchanged (still hardcodes `'uint32'`). EcuC's `reference` + `function-name` arms unchanged. The 5 common arms go through the helper, which returns the same string the inline switch did.
