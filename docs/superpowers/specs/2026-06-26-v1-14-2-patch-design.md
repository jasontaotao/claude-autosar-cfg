## Background

v1.14.1 PATCH-G ship + Security review deferred 4 items to v1.14.2 PATCH-H
per [[claude-autosarcfg-v1-14-1-shipped]] § Backlog notes (deferred):

- **H1** — `BSW-SEC-003` is declared in `diagnostics.ts:30` (WARN for
  empty `<STD-INCLUDE>` SHORT-NAME) but never emitted as a `Diagnostic`.
  `bswmd.ts:901` pushes a free-form string warning instead, which
  bypasses the strict-mode upgrade path promised by the v1.14.1 spec
  (line 168: "`strict: true` (CLI flag) promotes `BSW-SEC-003` from
  WARN → ERROR").
- **H2** — `<STD-INCLUDES>/<STD-INCLUDE>` paths are parsed into
  `BswModuleDef.includes[]` (v1.14.1 PATCH-G G1) but the generator
  never consumes them. Only the cross-ref path
  (`bswmdIndex[ref.targetModule]?.moduleHeader` → `buildReferenceIncludes`)
  is rendered into the Cfg.h. A BSWMD with `<STD-INCLUDES><STD-INCLUDE>
<SHORT-NAME>Os/Os_Cfg.h</SHORT-NAME></STD-INCLUDE></STD-INCLUDES>`
  declares the include but the generator drops it on the floor.
- **H3** — EcuC uses leaf-only `walkContainers` while Mcu uses
  ancestry-aware `walkContainersWithAncestry` (v1.14.1 PATCH-G G3).
  Tracked explicitly in `ecuc.ts:234-240` as a "v1.14.2 follow-up".
  EcuC's `paramByPath` lookups also need ancestry-tracking before
  the two generators have consistent nested-container behavior.
- **H4** — `ecuc.snapshot.capture.test.ts:59` populates
  `bswmdIndex: new Map<string, unknown>()` (empty), while
  `ecuc.snapshot.test.ts:71-73` and the other runtime tests
  (`mcu.test.ts:161-163`, `refs-emit.test.ts:80-82`) all carry
  EcuC/Os/Mcu `moduleHeader` entries. The capture script would
  regenerate goldens that DO NOT match the runtime behavior, so any
  future H1/H2/H3 change would diverge silently.

## Goal

`BSW-SEC-003` is properly emitted as a `Diagnostic` (with strict-mode
upgrade path), BSWMD `<STD-INCLUDES>` paths flow into the generated
Cfg.h, EcuC nested-container emit matches Mcu's ancestry-aware shape,
and the capture test regenerates goldens that match runtime behavior.

## Architecture

### H1 — `BSW-SEC-003` as a proper Diagnostic

End-to-end data flow:

```
Stage 1: Parse (bswmd.ts)
  buildEcucModule (line 886-919)
    OLD: empty SHORT-NAME → push string warning,
         drop from includes[]
    NEW: empty SHORT-NAME → keep as '' in includes[],
         no string warning (the validator owns the channel)

Stage 2: Validate (pipeline.ts)
  validateModuleHeaderPaths(bswmdIndex) (modules/_shared.ts:138)
    for each (modName, modDef):
      moduleHeader non-undefined && !validateHeaderPath → BSW-SEC-002
      for each inc in modDef.includes:
        inc === ''                          → BSW-SEC-003 (NEW)
        !validateHeaderPath(inc)            → BSW-SEC-002

  strict-mode upgrade (pipeline.ts S5 promotes WARN→ERROR)
    existing wiring promotes BSW-SEC-003 alongside ECUC-GEN-INFO-001
    → 1-line check in the upgrade loop
```

Why preserve `''` in `includes[]` rather than add a side-channel
field: empty-string entries flow through the same SEC3 iteration
loop as other entries, so the validator needs no shape change
beyond one extra `if` branch. The trade-off — `includes[]` may now
contain `''` — is handled at the consumption site (H2 below filters
empty strings before `#include` emission, consistent with the
existing `if (inc) ...` pattern in the template).

### H2 — `<STD-INCLUDES>` emission in Cfg.h

End-to-end data flow:

```
Stage 3: Generate (ecuc.ts / mcu.ts)
  OLD: buildReferenceIncludes(references, bswmdIndex, refIncludes)
       → refIncludePaths → template {{#each includes}}
  NEW: buildSelfIncludes(selfModDef.includes, refIncludes)    ← NEW
         iterate modDef.includes, skip '', dedupe via Set
       buildReferenceIncludes(references, bswmdIndex, refIncludes)
         (unchanged, BSW-SEC-004 push unchanged)
       → selfIncludePaths ++ refIncludePaths
         → template {{#each includes}} (0 template change)
```

`buildSelfIncludes` is a 5-line helper added to `_shared.ts`:

```ts
export function buildSelfIncludes(
  selfIncludes: readonly string[] | undefined,
  existing: ReadonlySet<string>,
): readonly string[] {
  const seen = new Set<string>(existing);
  const out: string[] = [];
  for (const inc of selfIncludes ?? []) {
    if (inc === '' || seen.has(inc)) continue;
    if (!validateHeaderPath(inc)) continue; // already BSW-SEC-002 in validator
    out.push(inc);
    seen.add(inc);
  }
  return out;
}
```

Why a helper and not inline: keeps ecuc.ts / mcu.ts at their current
line count and matches the existing `buildReferenceIncludes` shape
(D-rev2 R3 — extract shared helpers, not duplicate).

### H3 — EcuC ancestry parity

End-to-end data flow:

```
ecuc.ts:241
  OLD: walkContainers(eDef.containers, (container) => {
         const cDef = container;
         const path = `${eDef.shortName}/${cDef.shortName}/${pDef.shortName}`;
         ...
       });
  NEW: walkContainersWithAncestry(
         eDef.containers, eDef.shortName,
         (container, ancestry) => {
           const cDef = container;
           const path = `${ancestry}/${pDef.shortName}`;
           ...
         });
```

The `paramByPath` lookup key changes from
`${eDef.shortName}/${cDef.shortName}` (leaf only) to `${ancestry}`
(full chain). For flat fixtures, the two are identical (1-level
ancestry). For nested fixtures (e.g. `EcuC/PartitionConfig/PartitionBuffer`),
the new shape is `EcuC/PartitionConfig/PartitionBuffer` and the old
shape was `EcuC/PartitionBuffer` — this is the intended fix.

Tests:

- `ecuc.test.ts` gets 2 new cases: nested container emits param with
  full ancestry path; leaf-only container emits same as before.
- Snapshot tests: `ecuc-expected/{PreCompile,Mixed,Refs}-1/EcuC_Cfg.h`
  unchanged for flat fixtures (all current EcuC fixtures are 1-level);
  any future nested fixture is captured correctly.

### H4 — capture test bswmdIndex parity

`ecuc.snapshot.capture.test.ts:53-65` currently does:

```ts
const tree = normalizeToTree(
  new Map([[ecucDef.shortName, ecucDef]]),
  new Map(),
);
return {
  variant: 'PreCompile',
  bswmdIndex: new Map<string, unknown>(),  // ← EMPTY
  ...
};
```

Fix: populate `bswmdIndex` with the same EcuC/Os moduleHeader entries
as `ecuc.snapshot.test.ts:71-73` (and the Refs case requires Os only).
Since the capture script writes goldens that the runtime tests compare
against, the two must be in sync.

```ts
bswmdIndex: new Map<string, unknown>([
  ['EcuC', { shortName: 'EcuC', moduleHeader: 'EcuC/EcuC_Cfg.h' }],
  ['Os', { shortName: 'Os', moduleHeader: 'Os/Os_Cfg.h' }],
]),
```

This is a 2-line diff. The capture test is gated behind
`RUN_SNAPSHOT_CAPTURE=1` so it doesn't run during normal `pnpm test`
— the runtime snapshot tests are the actual coverage.

## Error Handling

| Trigger                                                      | Severity                                 | Code          | Behavior                                                                                                     |
| ------------------------------------------------------------ | ---------------------------------------- | ------------- | ------------------------------------------------------------------------------------------------------------ |
| `<STD-INCLUDE>` has empty `<SHORT-NAME>`                     | **WARN** (→ **ERROR** if `strict: true`) | `BSW-SEC-003` | pipeline continues; S5 strict-mode upgrade promotes to ERROR                                                 |
| `<STD-INCLUDE>` fails SEC3 whitelist                         | **ERROR**                                | `BSW-SEC-002` | unchanged (H1; BSW-SEC-003 takes precedence over BSW-SEC-002 for the empty case to keep the message precise) |
| `ref.targetModule` loaded but `moduleHeader` undefined       | **ERROR**                                | `BSW-SEC-004` | unchanged                                                                                                    |
| H2: BSWMD `<STD-INCLUDES>` entry duplicates a cross-ref path | no-op                                    | —             | `Set` dedupes                                                                                                |

Note: BSW-SEC-003 (empty SHORT-NAME) takes precedence over BSW-SEC-002
(whitelist violation) for empty strings — the SEC3 whitelist
`^[A-Za-z0-9_./-]+$` rejects `''` with the generic "fails whitelist"
message, which is less specific than "empty SHORT-NAME". The
`if (inc === '')` branch fires first.
