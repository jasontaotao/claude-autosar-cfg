## Goal

Ship a v1.15.0 MINOR that:

1. Eliminates the Stage-2 BSW-SEC-004 copy-paste between
   `EcuCGenerator` and `McuGenerator` (B-1).
2. Moves the BSW-SEC-004 push from emit-time (Stage 2) to
   validate-time (Stage 1), so missing-target-header is
   reported alongside the rest of the BSWMD path errors
   instead of as a side-effect of running a generator (B-2).
3. Preserves the existing behavior: all 2456 baseline tests
   pass, no snapshot regen, no behavior change for existing
   fixtures.

B-3 (generator type-driven refactor) and B-4 (BSWMD full vendor
modeling) are explicit deferrals — see "Out of Scope" below.

## Architecture

### B-1 — `resolveIncludesForModule` extraction

**Current state** (verified at `main@74d9741`):

`EcuCGenerator.emit` (`ecuc.ts:318-346`) and
`McuGenerator.emit` (`mcu.ts:223-249`) carry an identical
6-line block that:

1. Allocates a fresh `Set<string>` for cross-helper dedup.
2. Resolves the BSWMD-supplied self-includes via
   `buildSelfIncludes` (v1.14.2 H2).
3. Re-seeds the Set with the self-include paths.
4. Resolves the cross-module reference includes via
   `buildReferenceIncludes` (v1.14.1 G2).
5. Iterates `references[]` and pushes BSW-SEC-004 for every
   ref whose `targetModule` is in the BSWMD index but lacks
   `moduleHeader`.
6. Returns `[...selfIncludePaths, ...refIncludePaths]` to
   the template caller.

Steps 1-4 + 6 are pure data assembly; step 5 is a diagnostic
push. B-1 extracts steps 1-4 + 6 into
`resolveIncludesForModule` and leaves step 5 inline (B-2
moves step 5 in a separate commit).

**New helper** (in `_shared.ts`):

```ts
export function resolveIncludesForModule(
  moduleShortName: string,
  references: readonly { readonly targetModule: string }[],
  bswmdIndex: ReadonlyMap<string, BswmdIndexForModuleHeaderPaths>,
): {
  readonly selfPaths: readonly string[];
  readonly refPaths: readonly string[];
} {
  const refIncludes = new Set<string>();
  const selfDef = bswmdIndex.get(moduleShortName);
  const selfPaths = buildSelfIncludes(selfDef?.includes, refIncludes);
  for (const inc of selfPaths) refIncludes.add(inc);
  const refPaths = buildReferenceIncludes(references, bswmdIndex, refIncludes);
  return { selfPaths, refPaths };
}
```

**Call site change** (EcuC + Mcu):

```ts
// Before
const refIncludes = new Set<string>();
const selfDef = ctx.bswmdIndex?.get(eDef.shortName) as BswmdIndexForModuleHeaderPaths | undefined;
const selfIncludePaths = buildSelfIncludes(selfDef?.includes, refIncludes);
for (const inc of selfIncludePaths) refIncludes.add(inc);
const refIncludePaths = buildReferenceIncludes(
  eVals.references ?? [],
  ctx.bswmdIndex as ReadonlyMap<string, BswmdIndexForModuleHeaderPaths>,
  refIncludes,
);
// ... inline BSW-SEC-004 push (removed in B-2) ...
includes: [...selfIncludePaths, ...refIncludePaths],

// After
const { selfPaths: selfIncludePaths, refPaths: refIncludePaths } =
  resolveIncludesForModule(
    eDef.shortName,
    eVals.references ?? [],
    ctx.bswmdIndex as ReadonlyMap<string, BswmdIndexForModuleHeaderPaths>,
  );
// ... inline BSW-SEC-004 push (removed in B-2) ...
includes: [...selfIncludePaths, ...refIncludePaths],
```

**Why a separate helper, not inlining `buildReferenceIncludes` calls**:
the v1.14.1 PATCH-G cross-helper dedup contract
(`buildReferenceIncludes` takes a caller-owned Set) requires
the Set to be threaded through. Inlining the two calls at
every site is what created the copy-paste. A single helper
restores the encapsulation without breaking the immutability
guarantees the existing `buildReferenceIncludes` /
`buildSelfIncludes` tests assert.

**Why return `{ selfPaths, refPaths }`, not a flat list**:
the emit-time template context feeds `[...selfPaths, ...refPaths]`
in that exact order (AUTOSAR convention — self-includes first,
cross-refs second). Returning the pair lets the call site
preserve the ordering decision while moving the dedup logic.

**Behavior preservation**: identical to the current 6-line
block, byte-for-byte. Existing `ecuc.test.ts` and `mcu.test.ts`
snapshot/header tests must pass unchanged.

### B-2 — BSW-SEC-004 → Stage 1

**Current state**:

`EcuCGenerator.emit` (`ecuc.ts:333-346`) and
`McuGenerator.emit` (`mcu.ts:236-249`) push BSW-SEC-004 inline
at emit time. Consequences:

- The diagnostic only fires for modules that actually have a
  generator — a missing-generator module (WARN
  `ECUC-GEN-002`) is silently exempt from BSW-SEC-004 even
  when its references would fail the check.
- The diagnostic is interleaved with generator output; a
  pipeline user reading the diagnostics stream sees it after
  the generator's Handlebars work, not alongside other
  Stage-1 BSWMD path errors (`BSW-SEC-002` /
  `BSW-SEC-003`).
- Stage 1's S6 early-break does not catch it: the BSW-SEC-004
  push happens inside `generator.emit`, which only runs when
  Stage 1 returns clean.

**Fix**: Move the push to a new Stage-1 validator
`validateRefTargetHeaders(bswmdIndex, ecucValues)`, called
from `pipeline.ts` immediately after `validateModuleHeaderPaths`.
The Stage-1 S6 early-break then catches BSW-SEC-004 along
with the rest of the BSWMD path errors.

**New validator** (in `_shared.ts`):

```ts
export function validateRefTargetHeaders(
  bswmdIndex: ReadonlyMap<string, BswmdIndexForModuleHeaderPaths>,
  ecucValues: ReadonlyMap<string, EcucModuleConfigurationValuesInput>,
): readonly Diagnostic[] {
  const out: Diagnostic[] = [];
  for (const [moduleShortName, values] of ecucValues) {
    for (const ref of values.references ?? []) {
      const targetDef = bswmdIndex.get(ref.targetModule);
      if (targetDef && targetDef.moduleHeader === undefined) {
        out.push({
          severity: DiagnosticSeverity.ERROR,
          code: DiagnosticCode.BSW_SEC_MISSING_TARGET_HEADER,
          moduleShortName,
          ecucPath: ref.path,
          message: `Reference target module ${ref.targetModule} is loaded but its BSWMD omits <HEADER>; cannot auto-#include for ${ref.path}`,
        });
      }
    }
  }
  return out;
}
```

**Pipeline wiring** (`pipeline.ts`):

```ts
diagnostics.push(...validateModuleHeaderPaths(...));
diagnostics.push(
  ...validateRefTargetHeaders(
    args.bswmdIndex as BswmdIndexForModuleHeaderPaths,
    args.ecucValues as Parameters<typeof validateRefTargetHeaders>[1],
  ),
);
```

**Emit-site removal** (EcuC + Mcu):

```ts
// DELETED (after B-1 extraction):
for (const ref of eVals.references ?? []) {
  const targetDef = ctx.bswmdIndex?.get(ref.targetModule) as BswmdIndexForModuleHeaderPaths | undefined;
  if (targetDef && targetDef.moduleHeader === undefined) {
    ctx.diagnostics.push({ ... BSW-SEC-004 push ... });
  }
}
```

**Behavior preservation**:

- The 4 existing `bsw-sec-004.test.ts` cases (EcuC push,
  EcuC no-push, Mcu push, Mcu no-push) must still pass.
  The test still calls `g.emit(def, vals, ctx)` and asserts
  the diagnostic is present; with the push moved to Stage
  1, the test must be updated to call the new validator
  directly OR to call `runPipeline` and assert on the
  pipeline output. **Preferred**: call the validator
  directly (Stage-1 unit test); keep the emit-time tests as
  a no-op (the inline push is gone, but the existing test
  must be re-shaped to validate the Stage 1 surface).
- The 2 Refs-1 snapshot tests in `ecuc.snapshot.test.ts` and
  `mcu.test.ts` must still pass (the snapshot's BSWMD has
  `Os.moduleHeader = 'Os/Os_Cfg.h'`, so BSW-SEC-004 never
  fires; the validator returns an empty array).

**Why a separate validator, not a new arm of
`validateModuleHeaderPaths`**:

- `validateModuleHeaderPaths` is BSWMD-only (Stage 1 input:
  BSWMD index). The BSW-SEC-004 check needs ECUC values
  too, mixing two different pipeline inputs.
- The two checks surface different error classes: 002/003
  are BSWMD-author errors (fix the BSWMD); 004 is a
  project-configuration error (the ECUC values reference a
  target whose BSWMD is missing `<HEADER>`, fix the BSWMD
  or pick a different target).
- Separate validator matches the existing pipeline shape
  (every check is a separate `validate*` function with a
  distinct input signature).

**Order in pipeline.ts**:
`validateModuleHeaderPaths` runs first (BSW-SEC-002/003).
`validateRefTargetHeaders` runs after. This order matters
only in the rare case where both fire: BSW-SEC-002 fires
for the BSWMD's own `moduleHeader` invalid, BSW-SEC-004
fires for cross-module refs. The two are independent; the
order affects only the diagnostic-array order, which is
not user-visible.
